# Decisions & Assumptions

Running log of every non-default choice. Newest last.

## Stack (per spec defaults)
- **Tauri v2** (not Electron): lighter, OS-keychain access, locked-down webview, and the mobile path (iOS/Android) stays open with one codebase.
- React + TS + Vite + Zustand + **Tailwind v4** (CSS-variable tokens make dark/light a token swap; v4's Vite plugin removes the PostCSS config).
- SQLite via `rusqlite` (bundled, includes FTS5) — no external SQLite install.
- `keyring` crate → Windows Credential Manager. Chosen over `tauri-plugin-stronghold` for zero-config UX and true OS-level storage.

## Product interpretations
1. **Split counts = total conversations** in the split (not unread, not raw message count) — that's what makes a split read like a to-do list; matches Superhuman.
2. **Split matching**: first split whose rules match (settings order) claims the thread; the builtin "Other" (empty rule set) is the catch-all. Custom splits are inserted before "Other" so they win over it but not over Important/Calendar unless reordered.
3. **"Mark Done" = remove Gmail INBOX label** (archive). A reply re-adds INBOX server-side → the thread returns automatically on the next sync pass.
4. **Snooze** is local-first: archive remotely + `snoozed_until` locally; a 30s background tick returns due threads (re-adding INBOX+UNREAD remotely). A new reply wakes the thread early.
5. **Instant Reply insertion**: `Tab` previews suggestions in the thread view; `R`/`Enter` (reply / reply-all) inserts the highlighted suggestion for review. Nothing sends without an explicit `Ctrl+Enter`/Send click.
6. **`?` (Ask AI)** at P0 is thread-scoped Q&A (streams an answer grounded in the open thread + attachments). Inbox-wide Ask AI is P1 as specced.
7. **`Enter` in list opens** the conversation; in thread it's Reply-All (spec's dual meaning).
8. **Weekly streak** = `daily / 7` — consecutive zero-days roll into weeks; simplest model that matches "empty on consecutive days".
9. **Mock/demo mode**: with no account connected, both the browser build and the desktop app run on a seeded fixture inbox (same dataset in TS and Rust) so every feature is testable before credentials exist. Sent mail in demo mode appends locally.

## Technical choices
10. **DB is the UI's source of truth**; Gmail mutations are applied remotely first, then locally, so failures surface immediately and sync can't fight the UI.
11. **Sync = polling** (100 inbox + 50 recent archived threads, `historyId`-diffed) every 60s. Gmail push (Pub/Sub watch) needs a server — out of local-first scope; documented for P1 reconsideration.
12. **OAuth scope `gmail.modify`** — least privilege covering read/send/label. Not `gmail.readonly` + `gmail.send` because archive/label writes are core.
13. **AI streaming over Tauri `Channel`**; cancellation via per-request `AtomicBool` (frontend also stops listening immediately).
14. **Context budget in characters** (~4 chars/token heuristic, ≈25k tokens total). Truncation order: oldest messages' quoted trails stripped first, then oldest bodies capped, then attachment extracts capped per-file and in total. Newest message always survives intact.
15. **NIM is text-only** at P0 (vision support varies per NIM model); Claude/OpenAI get image attachments as base64 blocks. Attachment text extraction still applies to all three.
16. **.docx extraction is best-effort** (unzip → strip `document.xml` tags) per spec wording.
17. **Custom labels sync to Gmail** (name→id resolution, auto-create). Only system labels are assumed to pre-exist.
18. **Celebration images**: 4 bundled SVG landscapes ship in `public/inbox-zero/` (served from the app bundle — the spec's `assets/` folder would not be web-servable in Vite); a user folder is supported via the asset protocol, scope-limited to exactly that folder at runtime.
19. **Icons** are generated from a scripted PNG (`scripts/make-icon.mjs`) → `tauri icon`, so the repo has no unexplained binaries.
20. **Search** is FTS5 with sanitized prefix terms (alphanumerics only) — raw user input can't produce FTS syntax errors.
21. **No `.env` at runtime** — all credentials enter via the Settings UI into the keychain. `.env.example` documents what you need, nothing reads it.
22. **Repo layout deviation**: celebration assets live in `public/inbox-zero/` (not `assets/`) for bundling reasons; `tests/` holds the acceptance script pointer rather than a unit-test suite at P0 (the smoke-test checklist in SHORTCUTS.md is the verification vehicle, executed on Windows).

## v0.2 — Superhuman keymap parity, multi-account, signatures (2026-07-01)

Superhuman keys deliberately not mapped yet (features don't exist): `!` spam, `M` mute, `X` selection, `Z` undo, `G-S/D/T` views, calendar keys — listed in SHORTCUTS.md. `?` stays Ask AI per the original spec (Superhuman uses it for shortcut help); remappable anyway.
24. **Multi-account**: threads carry `account_id`; one `GmailSession` per account keyed by email; one shared OAuth client serves all Gmail accounts; refresh tokens per account (`gmail:refresh_token:<email>`), v0.1's single-token name kept as a read fallback. Slots = list order in Settings; `Ctrl+N` switches; splits/KB/AI settings are global across accounts (per-account splits deferred until someone actually wants them).
25. **Outlook (Microsoft Graph) is scaffolded, not implemented** — provider field, UI stub, and Azure app-registration docs exist; the Graph adapter is the headline of v0.3. Building it properly (folders ↔ archive semantics, conversation threading, MIME) deserves its own release rather than a rushed appendix to this one.
26. **Trash (`#`)** uses Gmail's `threads.trash` (30-day recovery server-side); locally the thread is dropped from the cache entirely. No trash view at P0 — Gmail's web UI covers recovery; no undo yet (with Superhuman parity, `Z` undo is the natural v0.3 companion to spam/mute).
27. **Signatures are per-account** (`settings.signatures[email]`), shown as a distinct block in compose and appended between body and quote on send. Not injected into the editable body so the AI draft flow can't accidentally treat the signature as draft text.
28. **Demo mode now has two accounts** (`demo@` + `angel@fission.local`) so account switching is demonstrable with zero credentials, in both the browser mock and the desktop fixture DB.
29. **Default AI = NIM with `deepseek-ai/deepseek-v4-pro`** — supersedes the original config's Claude default because the user supplied a NIM key and asked for DeepSeek v4 (verified live: model listed + streaming works). Claude/OpenAI remain one radio-click away.
30. **`has_key` is computed from the keychain on every settings read**, not persisted — keys seeded or revoked outside the app (e.g. via `dev-secrets`) always display truthfully.
31. **`dev-secrets` bin** seeds/checks/deletes keychain entries for development; values pass via env var only. Added `default-run = "fission-mail"` so `tauri dev` still resolves.
32. **Linux (Fedora) is a declared target**: keyring grew the Secret Service backend, CI gained an ubuntu `cargo check` with WebKitGTK deps, SETUP documents Fedora/Debian packages. Full Linux smoke test gates calling it "supported".
33. **Settings merge on read**: saved settings gain new default shortcuts/providers on upgrade instead of being wiped or frozen.

## v0.3 — Undo, Undo Send/Send Later, spam/mute, starred, snippets, unsubscribe, light theme (2026-07-01)
34. **Undo (`Z`) is a frontend inverse-action stack**, not an event-sourced log: each triage action pushes its inverse (done↔not-done, trash/spam→restore, mute→unmute, snooze→inbox, star↔unstar). Local-first and honest — no fake "undo" that leaves server state changed.
35. **Trash/spam became soft-hides** (`threads.hidden` column) so undo restores instantly from the local row; server-side they use `threads.trash` / SPAM labels, and restore uses `untrash`/label-removal + INBOX re-add. Sync can't resurrect hidden threads (inbox/done queries exclude trash+spam server-side too).
36. **Undo Send = a 10-second outbox fuse.** Sends are queued in a SQLite `outbox` table and flushed by a 3s background task; `Z` inside the window reclaims the draft into compose. **Send Later** is the same mechanism with a user-picked timestamp — it survives app restarts, needs no server. Failures retry with a 5-attempt cap. Trade-off (documented): a scheduled send only leaves the machine while the app is running.
37. **Mute** = custom `Muted` label + archive; a sync pass re-archives any muted thread that resurfaces with new replies. Matches Gmail's mute semantics without server rules.
38. **Unsubscribe (`Ctrl+U`)** reads the RFC-2369 `List-Unsubscribe` header captured at sync: https targets open in the browser, mailto targets prefill a compose. No List-Unsubscribe → clear toast. One-click POST (RFC 8058) is a possible refinement.
39. **Starred view** (`G S`) lists starred threads across inbox+done for the active account. Drafts/Sent views deferred (drafts aren't synced yet; sent threads need a `q=in:sent` sync pass).
40. **Light theme** is a token-set swap on `html[data-theme]` — tuned, not inverted (deeper accents, `--on-accent` flips to white). Dark stays default per the original spec.
41. Outlook's Graph adapter moves to **v0.4** — v0.3 spent its budget on triage parity, which benefits every account type.

## v0.4 — Distribution: auto-update, release CI, signing path (2026-07-01)
42. **Updates ship via GitHub Releases + `tauri-plugin-updater`** — no server, matching local-first. `latest.json` at `releases/latest/download/` is the feed; the app checks at boot + every 4h, downloads in the background, and offers "Update ready — Restart". Updater artifacts are minisign-signed; the private key lives in `C:\Users\swarn\.fission-mail-keys\` and the `TAURI_SIGNING_PRIVATE_KEY` repo secret (losing it orphans every install — it must be backed up).
43. **Release CI = tag-triggered `tauri-action`** on `windows-latest`; the existing `build.yml` stays as the PR/push gate. Installers are **unsigned for now**: Authenticode via Azure Trusted Signing ($9.99/mo) is the documented path (docs/DISTRIBUTION.md) and needs an account only the user can create; CI wiring is written down, not faked.
44. **NSIS `installMode: passive`** for updates — the updater runs the new installer with a progress bar and no wizard clicks, closest to "it just updated itself".

## v0.5 — Real-mail robustness: HTML rendering, attachments, drafts, incremental sync (2026-07-01)
45. **HTML mail renders sanitized-on-read** (ammonia allowlist in `mail/render.rs`; raw HTML stays in the DB so the allowlist can evolve without resync). Rendering happens in a script-less `sandbox="allow-same-origin"` iframe on a white "paper" card — dependable for real newsletters that assume a light background; height tracks content via ResizeObserver; links open in the system browser. Remote images are allowed (tracking-pixel tradeoff, same call Superhuman makes); `cid:` inline images resolve to data: URIs, fetched once (≤2 MB) and cached in `attachments.data`.
46. **Attachments**: bytes are fetched lazily (open/save actions), cached ≤25 MB in SQLite; compose attaches via a native file input (base64 over IPC, so the restart-surviving outbox keeps working). `build_rfc822` grew multipart/mixed + multipart/alternative.
47. **Drafts are local-only** (`drafts` table, opaque JSON payload). Esc = save + toast, Discard = delete, autosave every 800ms, `G D` resumes. Gmail-side draft sync deferred — local-first covers the "don't lose my work" promise without a sync headache.
48. **Sync is history.list-first**: per-account `history:<email>` baseline in kv; incremental pass refetches only affected threads (any thread, not just the newest 100); 404 (expired id) falls back to the old reconcile pass, now paginated (inbox 500 / done 200 threads). Initial backfill of 500 threads takes a minute of sequential fetches — acceptable one-time cost, documented here instead of hidden.
49. **Threading UX**: older messages collapse to one-line headers (avatar + snippet), last + unread stay open, quoted trails (`On … wrote:`, `gmail_quote`, trailing blockquotes) hide behind `•••` in both text and HTML modes.

## v0.6 — First-run experience + reported bugs (2026-07-02)
50. **v0.5.0 was not released separately** — item-3 code was already on the branch before item-2's smoke pass finished, and tagging mid-branch would have shipped updater artifacts whose self-reported version (0.4.0) disagreed with the tag. v0.5+v0.6 ship together as v0.6.0; the release train resumes one-tag-per-item after this.
51. **Onboarding is a 4-step gate** (connect → AI key → theme → 10-key tour) on `settings.onboarded`; replayable from Settings → Appearance. Existing upgraders see it once — that's intended, it doubles as the "what's new" tour.
52. **Account switching is Alt+1–9** (user preference; Ctrl+N kept free for browser muscle memory). Saved settings that still carry the old default (mod+N) migrate on read; custom remaps survive.
53. **A beta Google OAuth client can be baked in at build time** (`FISSION_GMAIL_CLIENT_ID/SECRET` env → `option_env!`), so testers never open the Google console. Desktop clients can't keep secrets by design (PKCE still guards the flow; Thunderbird et al. ship theirs the same way). Keychain-pasted credentials always win. CI reads them from repo secrets; `scripts/set-oauth-secrets.ps1` pushes them from the local keychain (run once by the owner).
54. **OAuth scopes grew to `gmail.modify openid email profile calendar.readonly`** in one step so each tester consents exactly once. Accounts connected before v0.6 need a Disconnect → Connect to grant the new scopes (profile photo + calendar panel appear after reconsent).
55. **Signatures are sanitized rich HTML** (paste straight from Gmail's signature box, images inline as data: URIs, ≤500 KB per image). Plain-text signatures keep the old all-text send path; rich ones add a multipart/alternative HTML body. Sanitization happens at save AND render (pasted rich text is untrusted).
56. **Sender avatars are deterministic monograms** (hue from email); the account's own photo comes from the OpenID userinfo endpoint, cached as a data: URI, overridable in Settings. Other people's Google photos need People-API contacts scope — declined (more consent friction than the feature is worth at beta).
57. **New-mail notifications** fire from the 30s background loop for unread-in-inbox threads newer than a boot-time watermark, only while the window is unfocused; toggle in Settings → Appearance. Window size/position persists via tauri-plugin-window-state.
58. **Vite ignores `src-tauri/**`** — Rust saves no longer full-reload the webview (bit the smoke-test agent; would bite anyone running browser + Rust dev side by side).

## v0.8 — Real-Gmail bug fixes, reliable auto-update, rename to Fission Mail (2026-07-02)
64. **Real HTML mail rendered as plain text because `body_html` was frozen NULL** (bug 1). Root cause: `upsert_thread`'s messages `ON CONFLICT` updated only `unread`; now it refreshes content columns, healing rows an older build froze (Gmail bodies are immutable, so a non-empty re-parse is always correct — guarded with CASE/COALESCE so a good body is never clobbered by an empty one). Also: base64url decode is now padding-indifferent (the old NO_PAD engine silently dropped padded bodies), and large HTML bodies stored behind `body.attachmentId` are fetched and inlined before parsing. Fix-1 only heals on refetch, so a **Repair Mail** palette command (`resync_account`) resets the historyId baseline to force a reconcile.
65. **Input lag was the trailing `refresh()` (4 IPC) after every triage** (bug 2). Triage store actions keep the optimistic move and drop the refresh; the backend reconciles via a **debounced** `mail:updated`. Backend triage is now local-write-then-spawn-remote (returns instantly), keyboard bindings are memoized (were rebuilt every keydown), and `get_thread` returns local messages immediately with UNREAD-removal + inline-image fetches backgrounded.
66. **Two-way sync could pin a stale historyId and miss server-side removals** (bug 3). incremental refetch errors now log-and-continue so the baseline advances; 400 (not just 404) falls back to reconcile; a forced full reconcile runs on startup and every ~10 min as the inbox-removal safety net; sync cadence raised to 30s. Triage with no live session returns an explicit "account not connected" instead of a silent local-only change.
67. **The shortcut-hint footer is toggleable** (bug 4) via a palette command, persisted in `settings.showShortcutBar` (default ON, `default_true` in Rust so older state defaults ON) — mirrors the theme/calendar toggle precedent.
68. **Blank message bodies recover on open** (bug 5): `refetch_message_body` re-fetches a thread whose body is empty; MessageCard shows "Loading message…"; opening a thread sets the id immediately so J/K is non-blocking. Adjacent-thread prefetch was skipped — bodies are already local after sync.
69. **"Calendar connection error" was a mislabeled API-disabled 403** (bug 6). gmail.rs now preserves the 403 body; calendar.rs classifies it (API-not-enabled vs. missing scope vs. generic) into actionable guidance; the granted OAuth scope is captured (warns at connect if calendar was skipped); `disconnect_account` revokes server-side for true re-consent. Enabling the Google Calendar API is now a required console step in the docs — the project owner does it once for all testers.
70. **Auto-update failures were invisible** (requirement A). Every check outcome now sets a user-facing status (latest / downloading / ready / failed-retry), surfaced in the header, a Ctrl+K command, and Settings → Appearance. Checks also run on window focus (not just boot + 4h). NSIS `installMode` set explicitly to `currentUser` so the silent updater never needs UAC.
71. **Renamed ZenBox Mail → Fission Mail** (repo `email-productivity-client`). Full rename incl. bundle identifier, DB filename, keychain service, crate/lib names, `ZENBOX_*` build env, and demo domains. Migration preserves existing installs: startup copies the old `zenbox.db` from the previous identifier's app-data dir, and keychain secrets migrate lazily via a legacy-service fallback. **The updater pubkey/signing key is unchanged**, so an installed ZenBox build verifies and auto-updates into Fission Mail. The baked OAuth client env prefers `FISSION_*` but falls back to `ZENBOX_*`, and CI maps the existing repo secret, so the shared beta client keeps working without renaming repo secrets. **⚠️ Updater continuity caveat:** existing v0.7 (ZenBox) binaries have the OLD `github.com/swarnavspujari/zenbox-mail/.../latest.json` endpoint compiled in, while v0.8 publishes to `email-productivity-client`. Their auto-update into Fission relies on GitHub's permanent redirect for the renamed repo (GitHub serves these for release-download URLs). Verify out-of-band after publishing — `curl -IL` the old endpoint and confirm it 30x-redirects to a valid `latest.json` — and do NOT recreate a repo named `zenbox-mail`, which would break the redirect and strand every existing install.

## v0.9 — Design system, inbox management, local-first calendar, Unsplash, Harper (2026-07-03)
72. **The Fission Mail Design System package is the visual source of truth.** theme.css is a straight port of its tokens: midnight-navy surface ramp `#000814→#002a52`, electric-violet accent, warm-gold highlight (`--accent-2`, never on buttons), six theme-aware tag hues, four cycling sender dots, and the bottom wash. The Tailwind `@theme` contract survived the retint, so most components changed zero color code. Fission Command stays dark on ANY theme via dedicated `--palette-*` tokens that the light theme deliberately doesn't override. The real brand icon replaced the hand-drawn "Z" tile (tauri icon set regenerated from the 1056px mark; 64px copy in `public/` for the header).
73. **List navigation is list-only now** (Bug 1): `list.next/prev` gate on `inList()`, and the v0.7 follow-into-thread branch is gone — `J`/`K`/arrows are inert while reading, `Space` (new `thread.scrollDown`) pages the open message. Instant, not smooth, scrolling: keyboard paging should feel like Superhuman, and smooth-scroll also stalls in headless webviews.
74. **Command failures are never silent**: both dispatch points (key bindings and the palette) route through `runCommand`, which toasts any rejection. The mail store's `archive`/`hide` also refresh-from-DB and rethrow when the backend rejects synchronously (account not connected) — the only failure path v0.8's triage:error reconcile didn't cover (Bug 4).
75. **Multi-select is a Set in the mail store** (Bug 5): `Ctrl+A` selects from the cursor down (per the spec, not whole-list), `X` toggles a row, Ctrl/Cmd-click too. Bulk Done/Trash/Restore/Label loop the existing per-id optimistic actions and push ONE undo entry (the ZeroSweep pattern), so a single `Z` restores the sweep. Esc clears the selection before anything else.
76. **Trash is a real, synced view** (Bug 3): `trash` joined MailView (rows where `hidden='trash'`), `Delete`/`Backspace` joined `#` (saved `#` bindings migrate on read, both cores), `G`,`T` navigates, `Shift+E` restores. Reconcile gained an `in:trash` pass (cap 100): server-trashed threads hide locally; locally-trashed threads resurrect only when the server affirmatively lists them live again (absence from a capped trash listing is not proof), with the pre-listing snapshot guarding against mid-reconcile races. "Empty trash" deletions fall out via the existing refetch-404 path.
77. **Gmail label ids resolve to names at the seam** (Bug 8): a `labels` table persists each account's id→name map (refreshed every reconcile from a new `GmailSession::list_labels`), `refetch_thread` translates inbound ids before storing, and a one-time repair rewrites rows older syncs stored verbatim. Label-scoped listing rides the existing `list_threads` command as `label:<name>` (json_each, quote-safe) rather than a new command. The FolderSidebar toggle mirrors `calendarOpen` exactly (`settings.sidebarOpen`). The v0.8 Muted re-assert workaround stays — harmless belt-and-braces now.
78. **The reading pane is the design's thread screen** (Bug 9): subject as a 22px in-pane title, 760px centered measure, raised softened cards, ContactPanel (identity + FTS-driven mail history; the mock CRM button was left out — no CRM backend exists to add to). The HTML iframe bakes the current theme's token VALUES into its srcDoc (`getComputedStyle` at render; the `data-theme` attribute flips synchronously via a zustand subscription so the values read fresh) — sanitization and the script-less sandbox are untouched.
79. **Calendar mirrors the mail read/sync split** (Bugs 6-7): `list_events` is a pure SQLite read over a new `events` table; `refresh_calendar` spawns a throttled fetch (skipped when the cached window covers the range and is <2 min old) and the 30s loop refreshes a rolling -7d..+21d window every ~5 min, emitting `calendar:updated` (payload = classified error or null). The per-calendar fetch resolves the token once and hits all calendars concurrently. Day state lives in a shared day-keyed store so the panel paints instantly on reopen; `←`/`→` are gated on a two-value `focusRegion` (week view screen, or the open+focused day panel) so arrows are never swallowed in the list. The deferred-since-#62 week view shipped (`G`,`C`).
80. **Daily Unsplash photo on empty splits** (backend-only key): keychain `unsplash:access_key` (BYO via Settings → Appearance) overrides `option_env!(FISSION_UNSPLASH_ACCESS_KEY)`; the key never reaches the web bundle. Guidelines followed literally — hotlinked `urls.regular` with its ixid, blur-hash placeholder (dependency-free decoder), download_location fired once per photo on first show, utm-tagged attribution. 24h cache keeps metadata + bytes (offline data: URI fallback), stale-if-error, and a ≤50 req/hour token bucket guards the cap. ⚠️ Owner action: run `gh secret set FISSION_UNSPLASH_ACCESS_KEY` once (the build agent's permission gate blocked repo-secret writes) — release.yml already reads it; until then releases ship without the baked key and the local keychain copy covers dev builds.
81. **Harper checks compose locally** (harper-core, `concurrent` feature — LintGroup isn't Send without it): `lint_text` runs the once-built curated linter on the blocking pool and returns UTF-16 spans so the UI never re-maps offsets. The UI is a transparent-text backdrop mirroring the textarea (wavy `--danger` underlines), with a caret-in-span suggestion bar — apply-on-accept, never auto-replace. The browser mock lints against a small fixture list so the demo exercises the same flow.
82. **Demo fixtures grew user labels** (Deals/LPs on three threads, TS + Rust in parity) so label views and the sidebar have real content to show without a connected account.
83. **Recipient autocomplete is derived from synced mail, not the People API.** A `contacts` table (account_id, email PK, name, freq, last_seen) is filled from every message's sender + recipients as mail syncs (indexed only on first-seen messages so freq counts distinct messages, not re-syncs), with a one-time backfill in `store::open` (kv-gated `contacts_built_v1`) so existing installs get an index immediately after updating. `search_contacts(query)` ranks prefix-on-name/email first, then freq, then recency, excluding the account's own address. The To/Cc fields autocomplete the token straddling the caret and insert `Name <email>` (accepted by the send validator, which only checks for `@`). This deliberately avoids the Google Contacts/People-API scope (DECISIONS #56 declined it — more consent friction than it's worth) while covering the real use case: everyone you actually email. If true address-book sync is ever wanted, it's a new OAuth scope + owner re-consent decision.

## v0.10.0 — Rich-text (WYSIWYG) compose with TipTap (2026-07-04)
86. **The compose body is now a TipTap (ProseMirror) rich-text editor**, replacing the plain-text textarea + Harper backdrop. Bold/italic/underline, bullet + numbered lists, and links (Ctrl+K), rendered headless and styled entirely off the design tokens (`.fm-editor` in theme.css) so it tracks both themes; Fission Command stays dark. Rationale for TipTap over a hand-rolled contentEditable, recorded per the brief: Harper's spell-check underlines are a **decorations** problem — ProseMirror inline Decorations render over arbitrary spans and auto-rebase as the document mutates, which raw contentEditable can't do without fighting caret preservation and the deprecated execCommand. StarterKit supplies the formatting marks/nodes, so the effort went into the Harper integration and the send path, not reinventing bold/lists. TipTap's ~150 KB (gzipped) is a non-issue for a desktop app.
    - **Harper is now a ProseMirror decoration plugin** (`src/features/compose/harper.ts`), replacing the textarea-backdrop `SpellCheck.tsx` (deleted — it couldn't survive a rich editor). On a debounced doc change it projects the doc to plain text while recording a position map (doc position ↔ plain-text offset — ProseMirror text offsets ARE UTF-16 code units, exactly what `lint_text` returns, so Harper's spans need no re-encoding; block boundaries and hard breaks become `\n` so word/sentence detection still works). It calls the **unchanged** `lint_text`, then sets one inline `Decoration` (wavy `--danger` underline) per finding via a `DecorationSet` that ProseMirror rebases through every subsequent transaction — underlines follow edits for free. A caret-in-range suggestion bar applies a fix as an ordinary transaction (`insertContentAt`/`deleteRange`) or dismisses per-finding; stale async lint results are dropped by a token counter plus a doc-equality guard.
    - **This resolves the v0.9.2 signature tradeoff (#85):** the rich/HTML signature is seeded into the editor body as real HTML (`signatureHtml()` in ui.ts) and renders + edits with its true formatting — bold, italics, links, and inline `data:` images (kept in the schema via the Image extension) — instead of being flattened to plain text. The `ComposeState.signature` field is gone (and `activeSignatureText()` with it); the signature now lives inside `body`, so nothing is appended at send.
    - **The send path is always rich now.** `outgoingFromCompose` sanitizes the editor HTML (`sanitizeUserHtml` — we now sanitize OUTBOUND too, mirroring the inbound ammonia pass), derives a **block-aware** plain-text fallback (a naive `textContent` ran paragraphs together), always sets `bodyHtml`, and appends the quoted history as a blockquote. Every message is therefore `multipart/alternative` (`build_rfc822` verified correct, unchanged). Paste is accepted and sanitized (`transformPastedHTML`), with ProseMirror's schema as the real guard — only known nodes/marks survive, so scripts never enter the doc.
    - **Keyboard-first coexistence with the global engine** (window keydown, `src/lib/keyboard.ts`): the editor's `handleKeyDown` returns `true` for Ctrl(/⇧)+Enter so ProseMirror's HardBreak keymap doesn't insert a newline, but does **not** `stopPropagation`, so the event bubbles to the global Send / Send-&-Done. Ctrl+K `stopPropagation`s (and opens the inline link bar) so the `mod+k` command palette can't open over compose. Ctrl+B/I/U + list toggles are StarterKit/Underline; Esc still bubbles to the draft-saving `back` command. Toolbar buttons are `tabIndex=-1` so Tab stays To→Cc→Subject→body. AI drafts (Ctrl+J) stream into the editor (`setContent` per chunk). Legacy plain-text draft bodies (saved pre-v0.10) convert to paragraphs on resume so their line breaks survive.

## v0.15.0 — Google Drive attachments + Contacts autocomplete + scope expansion (2026-07-07)

(Housekeeping: v0.11–v0.14 shipped without DECISIONS entries; their choices live in the release commit messages. Also note those releases were tagged from a side branch and never merged to `main` — this build re-baselined onto v0.14.0 and records that in the plan doc.)

87. **One fixed OAuth scope block, one re-consent.** The SCOPE constant grew to gmail.modify + gmail.settings.basic + openid/email/profile + **full `calendar`** (replacing calendar.readonly — groundwork for Build B's event CRUD) + **`drive`** + contacts.readonly + contacts.other.readonly + directory.readonly. There is no incremental consent: existing users Disconnect → Connect once (revoke-first keeps the re-grant clean, #69). Every feature gates on the **persisted per-account grant** (`granted_scopes:{email}` kv, exposed as `get_capabilities`): Google's per-scope consent checkboxes mean any scope can be individually missing, and accounts connected before this build report `legacy_grant` — both get a one-time notice and a "Reconnect to grant new access" CTA in Settings → Account. Reverses #56/#83's People-API avoidance by owner decision (spec 2026-07-07).
88. **Oversized upload transport: chunked raw invoke bodies, no Channel, no dialog plugin.** The webview slices the File into 4 MiB chunks (multiple of Google's required 256 KiB) and invokes `drive_upload_chunk` per chunk with a **raw request body** (`tauri::ipc::Request` / `InvokeBody::Raw`; the upload id rides in a request header, since a raw-body invoke carries no JSON args — which also rules out the spec's sketched `Channel<DriveUploadProgress>`). Rust PUTs each chunk to the resumable session URI (`Content-Range`; 308 = continue, 200/201 = done, 404 = expired). Progress falls out of the chunk loop in the frontend — exact, ordered, bounded memory, zero base64 JSON. Rejected: tauri_plugin_dialog native picking (WebView2 Files expose no paths and it would fork the attach UX) and a single raw-body invoke (whole file in RAM, no progress).
89. **Drive link chips are body HTML riding an extended Link mark.** A chip is an email-safe inline-styled `<a data-drive-chip data-drive-name style=…>` (renders in any client — no CSS classes to strip). TipTap's schema was the real obstacle, not the sanitizer (`sanitizeUserHtml` is a blocklist and already passes inline style): `ChipLink = Link.extend()` whitelists `style`/`data-drive-chip`/`data-drive-name` so chips round-trip parse/serialize. The plain-text MIME alternative flattens chips to `name — url` on their own line (`htmlToText`). Chips insert at the caret via a `fission:insert-html` window event to the live editor instance — writing to `compose.body` cannot work post-v0.10 (the editor is uncontrolled after seeding; SnippetPicker still uses the store route and doesn't render into the editor — pre-existing, out of scope here).
90. **Share-on-send semantics.** On send, chips still present in the body (deleted chips are skipped) raise one dialog: Share with recipients (view, per-address `permissions.create` with `sendNotificationEmail=false`) / Anyone with the link / Don't change access — remembered in `settings.driveShareMode`. Per-address failures toast and the **send proceeds** (Gmail parity). Sharing runs before `queueMail`, so Undo Send needs no special handling: undoing a send leaves files shared and uploads parked in the "Fission Mail Attachments" folder — accepted orphans, like Gmail's. Uploads in flight block send; closing the composer mid-upload abandons the pending chip (in-memory compose) and the file simply lands in the Drive folder.
91. **The three size gates stay coherent**: frontend 25 MB inline ceiling (now diverts to Drive instead of erroring — with a per-drop confirm or `driveAutoUpload: "always"`), Rust's 33 MB base64 validate_mail gate, and the outbox JSON payload are all untouched; Drive links bypass them by construction (no attachment bytes).
92. **People API sync is per-source replace, merged at query time.** `people_contacts` (account_id+source+email PK; sources contact/other/directory) is replaced per source on sync — and only for sources that fetched successfully, so a 403 (unchecked scope, consumer-account directory) never wipes previous rows. `search_contacts` merges the mail-derived history index with the People tables: prefix beats substring, then history (freq/recency — people you actually email) beats saved contacts beats other beats directory, dedup by email. Sync at connect, daily via the 5-minute background beat, manual "Refresh contacts" in Settings. Directory uses both DOMAIN_PROFILE and DOMAIN_CONTACT sources.
93. **Send-as is read-only this release** (`users.settings.sendAs` → kv `sendas:{email}`, listed in Settings → Account with default/verification/signature flags). The compose From-alias selector was spec'd as a stretch gated on full real-account verification of A0–A4; that verification is pending owner-side (see #94), so it wasn't built.
94. **Adversarial review round (24 findings confirmed by a 28-agent verify pass; 16 fixed, 4 accepted).** Fixed highlights: share-on-send was handing `"Name <email>"` tokens to `permissions.create` (silently failing for every autocompleted recipient) — addr-specs are now extracted first; a Drive upload finishing after its composer closed inserted its chip into whatever compose was open — uploads now bind to a `composeSeq` identity; the share/upload modals were invisible to the keyboard engine so Esc tore down the whole composer mid-send — capture-phase Esc + overlay gating; **Send Later now runs the same Drive gates as Send** (upload-in-flight block + share dialog) instead of silently skipping them; chunk PUTs retry 429/5xx with backoff, resync `sent` from the 308 `Range` high-water mark (short writes self-heal), and abandon the session server-side on unrecoverable failure; `replace_people` runs in one transaction (thousands of autocommit fsyncs under the global DB mutex froze all IPC for seconds); attach-as-copy enforces the aggregate 25 MB ceiling; the mock's `btoa` choked on em-dash fixture names (UTF-8-safe now); a missing `scope` field in Google's token response no longer misclassifies a fresh connect as a legacy grant; the 5-minute staleness beat no longer re-spawns doomed people syncs for accounts without a contacts grant; storage-quota 403s get their own guidance; `drive_folder` kv cleaned on disconnect; a Settings select can turn `driveAutoUpload: "always"` back off. ## v0.16.0 — Calendar two-way sync: event CRUD, invites, RSVP-from-mail (2026-07-07)

97. **The events cache table was dropped and recreated, not ALTERed** — the one deviation from the guarded-ALTER migration precedent (v0.2/v0.3/v0.5/v0.13), recorded per the spec's §9.2 open item. The full event model (description, htmlLink, etag, status, organizer, recurringEventId, hangoutLink, attendees JSON, calendarId + display name, accessRole, iCalUID) needed eleven new columns *and* a new PRIMARY KEY `(account_id, calendar_id, id)` — the old `(account_id, id)` can't key per-calendar sync upserts because the same Google event id exists on every attendee's calendar — and SQLite can't ALTER a primary key. Since `events` is a pure cache the next sync beat rebuilds, `open()` detects the old shape (no `calendar_id` in `pragma_table_info`) and drops it; stale `cal_window:` bookmarks are cleared with it. Reads dedup the same event id across calendars toward the most-privileged copy (owner > writer > reader) so shared calendars don't paint duplicates.
98. **Per-calendar syncToken engine under unchanged read shapes.** Each calendar syncs independently (kv `cal_sync:{email}:{calendarId}`): initial pass = windowed full fetch (-30d..+365d, `singleEvents=true`, paged) capturing `nextSyncToken`; every later pass sends only `syncToken` + `singleEvents` + paging (`orderBy`/`timeMin`/`timeMax` are invalid with a token), applying upserts and `status=cancelled` tombstone deletes; **410 Gone drops the token, clears ALL the calendar's rows (a deletion inside the token gap would otherwise zombie forever), and refetches the window**. `calendarList` accessRole/primary land on every cached event for UI gating; calendars that vanish from the list are pruned; per-calendar failures don't blank the rest (an error surfaces only when nothing synced). Cadence: the 5-min background beat (forced), UI-triggered `refresh_calendar` (now token-cheap, throttled to one pass/min), a new window-focus refresh, and a forced pull after every local write. `list_events`, `refresh_calendar`, and `calendar:updated` kept their exact shapes; only the internals changed. Disconnect now also clears the account's events + syncTokens so a reconnect starts clean.
99. **Writes are direct Google calls with If-Match etags — deliberately no offline queue (v1, per spec non-goal).** `create_event` / `update_event` / `delete_event` / `rsvp_event` send `If-Match` on update/delete; a 412 crosses IPC as `{status:"conflict", event:<fresh server copy>}`: the modal shows "This event changed elsewhere — review and retry" with Load-latest re-arming the form on the new etag; delete conflicts reopen the popover on the fresh copy. Success upserts the response locally, emits `calendar:updated`, and kicks an incremental sync. Guest notifications: any save/delete touching a guest list prompts Send / Don't send (→ `sendUpdates=all|none`); RSVP always notifies the organizer. RSVP patches the **full attendee array** with only the self row's `responseStatus` flipped — the attendees array overwrites wholesale, so a partial patch would drop the other guests — and update merges the cached attendee records back in for the same reason (bare emails would reset everyone's RSVP to needsAction).
100. **Calendar UI: modal + popover, Superhuman-keyed.** Click/drag on empty grid slots (30-min snap, plain click = 1h) opens the create modal prefilled; `B` = New Calendar Event (Superhuman's Create Event key — the spec's `c`-collision concern resolved by parity rather than a context chord); clicking an event opens a popover with time/location/Meet link/description/guest RSVP states. Affordances gate on the model: invited events (self attendee, organizer ≠ self) get RSVP Yes/Maybe/No; owned or guestless-writable events get Edit/Delete (with a notify-guests step when guests exist); reader/freeBusyReader calendars render as before with no edit surface. Declined events fade + strike through, pending invites render dashed. All-day events keep Google's exclusive-end date convention end-to-end (the modal displays the inclusive last day).
101. **Invite mail → RSVP bar via a captured ICS + iCalUID resolution.** Sync now captures the raw iCalendar payload (bare `text/calendar` part or invite.ics attachment — attachment-backed copies hydrate inline via the existing body-hydration pass) into `messages.ics_data` (guarded ALTER, 128 KB cap). A ~120-line tolerant reader (`mail/ics.rs`) pulls METHOD/UID/SUMMARY/ORGANIZER/DTSTART/DTEND/URL from the first VEVENT — deliberately not a full RFC5545 parser: no suitable iCal crate was already in the tree, and TZID-zoned times are approximated as local wall-clock because they only label the bar — **RSVP acts on the resolved Google event**, whose times are authoritative. Resolution: local `ical_uid` index first, then remote `events?iCalUID=` across calendars primary-first (no singleEvents, so a recurring invite resolves to its series master and one RSVP covers the series, like Gmail's own buttons). Unresolvable invites fall back to "Open in Google Calendar" (the ICS `URL:` when present); METHOD:CANCEL renders an informational strip. Invite mail synced before v0.16 has no captured ICS until its thread re-syncs (`ics_data` heals via COALESCE on upsert); the existing resync repair covers the backlog.
102. **Mock parity: full CRUD + RSVP over the fixture calendar, in both demos.** The browser MockBackend gained a persisted overlay (created / patched / deleted over the synthesized day fixtures) implementing create/update/delete/RSVP with real etag bumps — a stale etag returns the same `{status:"conflict"}` shape, so the 412 flow is demoable offline. The desktop demo seeds ICS payloads for the two "Invitation:" fixture threads (the board meeting resolves to that day's fixture event; the SaaStrix dinner deliberately doesn't, exercising the fallback) and records demo RSVPs in a kv overlay re-applied on read. Fixture events grew organizer/attendee data — the board meeting arrives as a pending invite so RSVP affordances are demoable everywhere.

103. **Review round (8-angle finder pass, 43 candidates; 24 fixed, the rest refuted or accepted).** Fixed highlights — *sync correctness:* a Google syncToken permanently encodes its initial fetch's time bounds, so the engine now re-anchors (drops tokens + refetches) every 30 days to roll the -30d..+365d horizon forward, and navigating outside the window triggers a one-off range fetch with the old coverage-bookmark throttle (without either, dates past the horizon were permanently blank); calendars removed from calendarList also drop their syncTokens (a re-selected calendar previously resumed a stale token against an empty cache and rendered almost nothing until a 410); page-cap overflow now caches the partial window instead of erroring every beat, and an over-cap incremental delta resets the token rather than failing forever. *Write safety:* POST (events.insert) is no longer retried on 5xx — a committed insert behind a 500 would duplicate the event and re-email every guest (no idempotency key exists in this API); 429 still retries. *Invites:* RSVP resolution ranks candidate copies by (own attendee row, writable calendar) so it can't target a read-only shared-calendar copy and 403; unresolvable UIDs get a 10-minute negative cache instead of re-scanning every calendar per thread open; `VALUE=DATE-TIME` no longer false-matches the date-only parser; all-day invites render as day ranges, not "12:00 AM – 12:00 AM". *Timezones:* all-day exclusive ends use calendar-day arithmetic (midnight+24h lands at 23:00 on a 25-hour DST day, collapsing start==end → Google 400), and zones that skip midnight (Chile-style DST) fall back to 01:00 instead of silently dropping the event. *Polish:* clicking an event keeps calendar keyboard focus (stopPropagation was eating the focus-region mousedown); a mock event edited onto another day no longer vanishes; `B` from the mail list creates today's event instead of a stale navigated day; demo RSVP dispatch keys off the account provider, not the client-supplied calendar id; pre-v0.16 demo DBs get their invite ICS healed in place; oversized .ics attachments are skipped before download; incremental deltas apply in one transaction (not per-row autocommits under the global DB mutex); the calendar cap rose 10→25 with primary/privilege priority; retry + error-shaping helpers unified; drag ghosts only re-render on slot changes. Accepted, not fixed (v1 scope): desktop-demo create/edit/delete error like the v0.15 Drive-in-demo precedent (the browser demo carries full CRUD parity; RSVP works in both); one calendarList round trip per write (writes are rare, staleness isn't); time-format/slot-drag/guest-parsing duplication across the new calendar components; InviteBar keeps component-local state; and the mock fixture mirror in two languages, inherent to the mock-parity architecture.
